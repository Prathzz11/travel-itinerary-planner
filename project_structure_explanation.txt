TRAVEL ITINERARY PLANNER - COMPREHENSIVE PROJECT EXPLANATION

This document provides a highly detailed explanation of every file and folder in the project. It is structured to be read as a comprehensive manual for your viva, detailing the exact role, purpose, and utility of every component that makes up your React Single Page Application. 

CONFIGURATION AND ROOT FILES

package.json and package-lock.json
These files serve as the absolute core of the Node.js project environment. The core package JSON file defines all the metadata about your web application, including its name, current version, and execution scripts such as the command you use to launch the development server. Furthermore, it strictly defines every third party dependency and open source library the project relies on, such as React, React Router, and Axios. The accompanied lock file ensures that exactly the same versions of these libraries are installed on any machine, granting deterministic and safe builds.

vite.config.js
This file governs the configuration for Vite, which is the extremely fast, native frontend build tool used to serve your project. This configuration allows Vite to parse your React JSX files, offer hot module replacement so the browser updates instantly when you press save, and ultimately bundle all of your scattered javascript files into optimized, compiled assets for production deployment.

index.html
This single HTML file is the foundation of your Single Page Application. When a user navigates to your website, this is the only HTML file the server ever provides. Inside the body tag, there is an empty div element with the ID of root. Once this file loads in the browser, your compiled JavaScript takes over entirely, rendering the entire Travel Itinerary Planner application dynamically into that single, empty div element.

.gitignore
The git ignore file tells your version control system exactly which files it should never upload or track. This is crucial for avoiding the upload of locally generated folders like the massive node_modules folder, or potentially sensitive local environment variables that you do not want hosted publicly on GitHub.

syllabus_analysis.md
The syllabus analysis file provides an academic breakdown of how the implemented features relate to specific coursework units and laboratory requirements.

PUBLIC ASSETS

public/bg-globe.mp4
This file is a localized high-definition video asset. It is placed in the public folder so that it can be directly referenced by URL without being altered by the bundler. It serves as a continuous, visually impressive animated background loop behind your web application to give the project a premium, travel-oriented aesthetic.

public/navigation.png and public/icons.svg
These scalable vector graphic files exist in the public directory to provide browser-level iconography. The favicon appears directly in the user's browser tab. Because they are in the public folder, the HTML document can load them instantly before the React application even begins to boot up.

MAIN REACT APPLICATION FILES (src folder)

src/main.jsx
This is the absolute genesis point of your React application. When the website boots up, this file searches the index.html file for the root div and attaches the React application to it. It imports your top-level CSS routines and wraps the core application inside strict mode, which doubly runs checks in development to catch silent rendering issues.

src/App.jsx
The App file is the structural backbone of your platform. It primarily establishes the React Router map, linking specific URL endpoints in the browser to their corresponding visual pages (for instance, linking the /login URL to the actual Login component). It also systematically wraps all of these pages inside all of your global context providers, ensuring that every page has access to shared memory, authentications, and global themes.

src/index.css
This standard stylesheet applies cascading rules across the entire application interface. It resets default browser behaviors, eliminating margins and padding on the body element. It also handles global imports for typography and provides standard styling to HTML baseline elements that every page will inherit.

STYLE AND ASSET FOLDERS

src/styles/variables.css
This dedicated styling file operates as your central design system repository. Rather than hardcoding colors and sizes thousands of times across the application, this file establishes global CSS custom properties. This makes it incredibly easy to enforce exact branding colors, standardize border radiuses, and makes implementing light or dark mode toggles completely seamless.

src/assets/hero.png, react.svg, vite.svg
These are static, visual media assets imported intrinsically into your React layouts. Unlike public assets, assets stored in this folder are hashed and aggressively aggressively minified by the Vite bundler during production builds to guarantee lighting-fast loading speeds for your end user.

CONTEXT PROVIDERS (src/contexts)

src/contexts/AuthContext.jsx
This context acts as the centralized authority representing the user's security and identity status. It remembers if the user is logged in, stores user preferences like display name and currency, and allows any component in the application to instantly determine exactly who is currently browsing the system.

src/contexts/TripContext.jsx
The Trip Context wraps global memory regarding the overarching trips a user is involved in. It maintains a persistent top-level list of upcoming destinations, the duration of those tours, the people who have been invited, and their respective permissions as viewers or editors.

src/contexts/ItineraryContext.jsx
Where the trip context handles top-level data, the Itinerary Context drills down into the specific, micro-level daily planning. It hosts state arrays to track individual events across travel days, mapping exact hours to specific flights, hotel check-ins, or dinner reservations, creating a timeline engine.

src/contexts/BookingContext.jsx
This context explicitly manages external documentation. When a user uploads a flight confirmation number or a hotel booking link, this state manager retains those records globally across the session so they can be parsed out and attached to multiple different days or summary views.

src/contexts/ExpenseContext.jsx
The Expense Context implements financial tracking. It is responsible for globally holding state associated with every penny spent, allowing the application to add up individual food and transport expenses to instantly project if the user is meeting their total target travel budget.

src/contexts/ExploreContext.jsx
This specific state provider manages the social, community-driven aspect of the application. It fetches and retains public itineraries created by other users in the ecosystem, caching them in memory so users can rapidly search, filter, and fork popular travel templates without lag.

src/contexts/NotificationContext.jsx
This provider wraps the entire application with an invisible alert mechanism. Subscribing to this context allows any component that experiences an error, or achieves a successful save, to instantly trigger a floating toast notification alert in the corner of the user's screen.

CUSTOM REACT HOOKS (src/hooks)

src/hooks/useAuth.js and src/hooks/useTrip.js
These custom hooks act as developer-friendly syntactic sugar. Rather than manually importing raw Context objects over and over, these helper functions reduce boilerplate overhead and instantly expose all required context variables to whichever component calls them.

src/hooks/useDebounce.js
The debounce hook is a performance optimization tool. If a user is rapidly typing in a search bar, this hook purposefully intercepts the keystrokes and waits for the user to pause for a fraction of a second before signaling the backend, drastically reducing unnecessary server stress and API calls.

src/hooks/useForm.js
This hook completely encapsulates the repetitive logic of managing form inputs. Whenever an application needs a user to type out information, this hook binds the text to state, validates complex business logic like making sure an email is properly formatted, and retains drafts across the platform.

src/hooks/useLocalStorage.js
This incredibly powerful memory hook perfectly mimics React's native state hook, with one monumental addition: it constantly synchronizes the data directly into the user's browser storage. This guarantees that if the user refreshes the page or their computer restarts, their itinerary progress instantly returns exactly as they left it.

src/components/explore/ReviewSection.jsx

These visual components prevent dead clicks. They ensure that while an asynchronous backend API save operation is in progress, buttons are locked down and spinners animate, providing crucial psychological feedback informing the user that the system is successfully thinking.

src/components/ui/NotificationContainer.jsx

BACKEND SERVER (server folder)

server/index.js
This file serves as the main entry point for the Express.js application backend. It initializes the web server, configures critical security middleware such as helmet and express-rate-limit to prevent abuse, establishes cross-origin resource sharing (CORS) rules for secure client communication, and connects to the underlying MongoDB database. Furthermore, it organizes and mounts all modular route handlers for different application components under the /api namespace.

server/package.json
The package file for the server dictates all Node.js backend dependencies required to successfully run the API. It ensures that critical third-party security libraries like bcryptjs for password hashing, jsonwebtoken for session management, and mongoose for database interactions are consistently installed on any execution environment.

server/config/db.js
This configuration file encapsulates the logic for establishing a stable connection to the remote or local MongoDB database using Mongoose. It ensures the server gracefully handles connection states and outputs clear error diagnostics if the database becomes unreachable.

server/models/
These files contain the strict Mongoose schema definitions mapping directly to the NoSQL database collections. Files like User.js, Trip.js, and Itinerary.js define the precise shape, validation rules, and relationships of the data saved within the system, ensuring data integrity and enforcing structured relationships between travelers, their trips, and their daily planned activities.

server/controllers/
This directory hosts the core business logic of the backend application. Files like authController.js or tripController.js act as the intermediary layer between the database models and incoming network requests. They handle all complex operations: validating incoming data, querying or updating the database, handling complex operations like trip member authorization, and sending the formatted JSON response back to the client interface.

server/routes/
These router files isolate all API endpoint definitions. Files like authRoutes.js or tripRoutes.js cleanly map specific incoming HTTP requests (GET, POST, PUT, DELETE) and URL paths (like /api/auth/login) to their respective functions in the controller layer. This separation of concerns allows developers to instantly discover all available API interactions without sorting through dense business logic.

API INTEGRATION AND USAGE

This project relies on two distinct API paradigms to function: a Third-Party External API ecosystem provided by Google, and an Internal Custom REST API serving as the application backend.

1. Third-Party API Integration (Google Maps Ecosystem)
The application extensively integrates the Google Maps external platform to provide real-time, interactive geographical data. This is managed via the "@react-google-maps/api" library and authenticated securely through a "VITE_GOOGLE_MAPS_API_KEY" environment variable. The integration spans three distinct Google services:
- Maps JavaScript API: Utilized primarily within "src/components/map/InteractiveMap.jsx". This API handles the physical rendering of the map canvas, allowing the application to dynamically plot custom destination markers, center coordinates on specific cities, and render interactive info-windows for travelers.
- Places API: Employed within "src/components/itinerary/ActivityModal.jsx" to power the Autocomplete search bar. When a user begins typing a location or restaurant name, this API queries Google's massive global database in real-time, returning verified addresses, auto-completing user text, and instantly resolving the location into exact latitude/longitude coordinates.
- Directions API: Called dynamically by the InteractiveMap component using "window.google.maps.DirectionsService". It algorithmically calculates physical routes between sequential itinerary stops, drawing visual paths across the map and estimating travel times based on specified transit or driving modes.

2. Internal Custom REST API (Express.js Backend)
The React frontend is completely decoupled from the database and relies entirely on HTTP requests to communicate with the Node.js backend.
- Axios HTTP Client: Located centrally at "src/services/api.js", Axios acts as the primary vehicle for all outbound internal API calls. It standardizes HTTP headers, injects JSON Web Tokens for authentication, and implements "axios-retry" which automatically catches failed network requests and gracefully re-attempts them using exponential back-off delays.
- Endpoints & Routing: The frontend targets the "VITE_API_URL" (e.g., http://localhost:5000/api), hitting dedicated routes configured by the Express server. The React application sends GET requests to fetch data (like retrieving public itineraries), POST requests to create entities (like signing up a user), and PUT/DELETE commands to alter existing database records. The entire relationship guarantees that the frontend never accesses the MongoDB database directly, maintaining strict architectural security.
